## 当前实现摘要

当前阅读模式入口和数据流大致如下：

- 文学区 fid 白名单：`49`, `55`, `60`, `350`。
- `getReadingStream(tid, authorid, page)` 使用 mobile API 的 `authorid` 参数，只取楼主楼层流。
- `buildReadingBook(first)` 的章节识别顺序：
  1. 只从首楼 blocks 中找带 `pid=` 的链接目录。
  2. 如果没有链接目录，再从楼主楼层文本的第一段识别标题。
  3. 如果仍没有章节，兜底把首楼作为单章短篇。
- `Reader` 首次只拉第 1 页楼主楼层；如果目录章节没有正文 blocks，则按需找对应 pid；如果不是目录型且总页数大于 1，则预拉剩余楼主页后重建 book。

## 已确认问题

### P0. 只要首楼出现任何可识别目录链接，就不会再用楼主楼层流补全

`buildReadingBook` 在首楼解析到 `chapterLinks(firstPost.blocks)` 后，会直接跳过楼层标题识别。结果是：

- 首楼目录不完整时，阅读模式只显示目录里已有的章节，漏掉目录未收录的新章节。
- 首楼有「临时目录」「更新到某处」「番外目录」等部分链接时，后续正文楼层不会进入目录。
- 首楼目录链接标题只要符合 `CHAPTER_RE` 就被认为是完整目录；没有校验目录数量和楼主实际正文楼层数量。

需要补救：目录不能作为唯一真相。应把「目录链接」和「楼主楼层标题扫描」合并，再按 pid 去重；如果目录数量明显少于楼主正文楼层，应提示或自动补全。

### P0. 多楼层连载如果楼层开头不是当前正则，会被识别成短篇

当前正则只识别：

- `第N话/話/章/节/節/回`
- `episode N`
- `番外`
- `序章`
- `终章/終章`
- `后日谈/後日談`

实际论坛标题可能出现：

- `第 N 篇`, `第 N 夜`, `第 N 则`, `第 N 幕`, `Act N`, `Chapter N`, `Part N`
- `01`, `001`, `1.`, `【一】`, `一、`, `其一`
- 日期式更新：`2026.6.16 更新`, `6/16`
- 作品标题式分段：只有小标题，没有「第 N 话」
- 翻译帖常见的 `番外 22.5`, `22.5话`, `interlude`, `extra`

如果楼主多楼层发布但每楼开头不匹配，`buildReadingBook` 最终会走首楼兜底，`shape` 变成「短篇」，后续楼层全部丢失。

需要补救：章节识别应由「标题正则」升级为「楼主楼层正文候选判定」：结合楼层长度、楼主连续更新、目录文本、标题样式、楼层间隔、是否包含作者碎碎念等特征。

### P0. 多楼层无章节标题的作品会只保留首楼

即使 `Reader` 会在非目录型长帖中预拉所有楼主页，`buildReadingBook` 仍要求后续楼层能识别出章节标题。若后续楼层是直接正文、图片正文、或只有短小说明，代码会回落为：

```ts
chapters = [{ id: firstPost.pid, pid: firstPost.pid, no: 1, title: first.title, blocks: firstPost.blocks }];
```

也就是「整帖短篇 = 只取首楼」，这和 `docs/05-READING-MODE.md` 里「小说正文 = 楼主本人的楼层流」的抽象冲突。

需要补救：没有章节标题时，也应把楼主每个正文楼层作为「未命名章节」或合并成连续章节，而不是丢弃 2 楼之后的楼主内容。

### P1. 纯文本目录没有被解析成目录

已知样本 `571449` 属于「首楼纯文本目录，无链接；后续楼层逐话发布」。当前实现不会解析纯文本目录，只能依赖后续楼层标题识别。

后果：

- 目录里的正式章节名无法用于补齐标题。
- 目录能说明章节顺序、番外编号、小数话编号，但当前全部丢失。
- 如果后续楼层标题不完整，阅读模式目录会出现泛化标题或短篇误判。

需要补救：解析首楼纯文本目录，提取行级章节名；把它作为楼主楼层标题识别的标题源和顺序校验源。

### P1. `chapterTitleFromBlocks` 只看每个 text block 的第一段

代码只取 text block 的第一个非空行作为候选标题。如果楼层开头有：

- `本帖最后由...编辑`
- `译者的话`
- `前情提要`
- 警告/授权/转载说明
- 图片或 quote 在正文前

真实章节标题可能出现在后面几行，但当前实现不会继续扫描。结果是正文楼层被漏识别。

需要补救：扫描前 N 个非空短行，过滤编辑标记、授权说明、译者注、警告语；必要时结合目录文本和楼层长度判断。

### P1. 作者碎碎念和公告楼层会被误当正文或误当章节

`docs/05-READING-MODE.md` 已指出楼主楼层里混有「作者碎碎念」。当前实现只做标题识别，不区分：

- 正文章节
- 更新公告
- 译者说明
- 请假/停更/补档
- 目录维护楼
- 评论回复/感谢楼

后果包括：

- 误生成「番外」「后日谈」等章节。
- 把非正文楼层作为阅读正文。
- 章节统计、进度和目录污染。

需要补救：正文楼层要有质量阈值和排除规则，例如正文长度、段落数量、标题词、是否大面积外链/公告语、是否包含「更新」「占楼」「目录」「授权」等。

### P1. 章节完整性缺少自检

当前没有任何地方比较：

- 识别到的章节数
- 楼主楼层数
- 首楼目录项数
- 标题里声明的更新进度，例如「更新至第420话」
- 回复/楼层总页数

因此「长篇被短篇化」「目录漏几十章」「只识别一半」不会触发明显错误，只会安静地进入阅读界面。

需要补救：构建 `ReadingBook` 时返回 diagnostics，例如 `opPostCount`, `linkedTocCount`, `titleDetectedCount`, `fallbackPostCount`, `declaredLatestNo`；当差异过大时提示用户并记录日志。

### P1. 阅读入口过宽，失败体验会集中爆发在阅读器内

`Thread.tsx` 对文学 fid 一律提供阅读模式入口。入口本身没有先判断是否像小说，也没有预估识别质量。

后果：

- 公告、求文、讨论、补档、资源帖也会出现阅读模式。
- 进入后可能显示短篇、空正文或错误。

需要补救：入口可以保留，但视觉权重和文案应依赖识别置信度。低置信度时用「尝试阅读模式」或收在更多菜单里。

### P2. 章末评论定位对楼层 position/分页假设较脆弱

评论页 hint 使用 `Math.ceil(chapter.pos / book.ppp)`。这里要求 `chapter.pos` 是未过滤整帖楼层位置，而 `book.ppp` 是未过滤页大小。如果 authorid 过滤后的 `position` 语义变化，page hint 会错；错后才依赖 findpost fallback。

需要补救：优先记录并使用目录 pid 的 findpost 解析结果；page hint 只作为优化，不作为主要定位依据。

### P2. 短篇内部章节细分仍未实现

`docs/05-READING-MODE.md` 写了 `571902` 这类无目录单楼帖可「内部按空行/标题再细分，并保留整楼当一章兜底」。当前实现只有整楼当一章，没有内部小节/标题拆分。

需要补救：单楼长文可做可选小节导航：识别明显分隔符、居中标题、数字标题、`---`/`====`/空行密集分段，但正文仍保持连续阅读。
