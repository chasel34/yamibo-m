# 小说阅读模式 · 调研与方案 (Reading Mode)

> 面向「小说/翻译」类帖子的沉浸式阅读体验，目标是把论坛"一楼一楼刷"的形态，
> 转成类似微信读书的「按章节阅读」。本文件记录调研结论、已定方向与待办。
> 接口模块对照 [02-API.md](02-API.md)，相关浏览增强见 [04-BACKLOG.md](04-BACKLOG.md)（P2「只看楼主 / 跳楼」）。
>
> 状态：**方向已定，未开发**。调研于 2026-06-14，基于已登录态对真实 mobile API 实测。

---

## 1. 背景

百合会文学/翻译区有大量小说帖，长度跨度极大：从单楼短篇，到上百话、上万楼回复的超长连载。
长篇在论坛形态下阅读困难（正文与读者回复交织、散落在几百页里）。需要一个**只读增强**功能，
把这类帖子转成沉浸式阅读视图。属只读 v1 范畴（纯浏览，不涉及写操作）。

## 2. 调研结论：这类帖子的真实结构

用已登录浏览器直接请求三个样本帖的 mobile JSON API（`module=viewthread`），结论如下。

| 帖子 tid | 规模 | 正文存放方式 | 目录 |
|---|---|---|---|
| 571902 P站翻译 | 3 回复 | 整篇在楼主首楼（~24KB 一个 post），无 `第N话`/`<hr>` 章节标记 | 无 |
| 571449 渣翻 | 168 回复 | 楼主逐楼连载：1 楼=简介+目录，2 楼=第1话，13 楼=第2话…中间夹读者回复 | 首楼**纯文本**目录（无链接）|
| 521519 长篇 | 16660 回复 / 420 话 | 楼主逐楼连载，散落在 ~833 页，与读者回复交织 | 首楼自动生成折叠目录，**423 个链接**，每个 `findpost&pid=` 指向具体楼层 |

**统一抽象：小说正文 = 楼主本人的楼层流；读者回复是噪音，不进入正文。**
单楼短篇只是"楼主只有 1 个楼层"的特例。

**内容保全优先原则**：阅读模式宁可多带少量楼主说明、目录维护、作者碎碎念，也不能漏掉可能的正文。
章节标题、目录解析和正文筛选都只是提升阅读体验的手段；当识别不确定时，必须降级为"完整但不够精致"，
而不是为了目录干净而丢弃楼主后续楼层。

### 决定可行性的关键技术点（均已实测验证）

1. **`authorid` 过滤在 mobile API 有效** ✅
   `?module=viewthread&tid=<tid>&authorid=<楼主id>` 把整帖塌缩成只剩楼主的楼层。
   长篇的 `thread.replies` 从 16660 变成 **489（楼主自己的楼层数）**，分页按楼主算：
   **约 25 页拿全 420 话，而非 833 页**。这是整个功能跑得起来的基石。
   （注：`opt=1` 无效，必须用 `authorid=`。）
2. **章节标题就嵌在每个楼主楼层开头**：`Episode 1`、`第1话 ……`、`episode 418 宫城的脑海中`，可正则识别做章节切分。
3. 楼主楼层里**混有少量"作者碎碎念"**（如"第二章开始视角会切换…""谢谢大家的支持…"），非正文章节，需识别/弱化。
4. **`pid → page` 可解析**：`forum.php?mod=redirect&goto=findpost&ptid=<tid>&pid=<pid>` 的最终 URL 带 `&page=N`。
   用于按目录链接定位章节所在页（不带 authorid 过滤时）。
5. 目录格式不统一（521519 带 `findpost` 链接、571449 纯文本无链接、571902 无目录），
   所以**不能只依赖目录**。目录是标题和定位信号，不是正文完整性的唯一真相。
6. 现有 `src/util.ts` 的 `parseMessage` 已能把楼层 HTML 转成 blocks；`src/history.ts` 已有本地存储范式（阅读进度复用）。

## 3. 已定方案

> 经讨论拍板，以下为确定方向。

### 3.1 数据 / 章节切分
- **楼主楼层流为正文基准，目录只作为增强信号**：
  - `authorid` 过滤后的楼主 posts 是阅读模式的正文候选全集。
  - 链接目录、纯文本目录、楼层标题扫描共同参与章节构建，不再三选一。
  - 目录链接可提供 `pid → title` 和按需加载定位，但不能让未出现在目录里的楼主正文楼层消失。
  - 纯文本目录可提供章节名和顺序校验，但不能替代楼主楼层正文。

- **章节构建采用合并模型**：
  1. 拉取楼主楼层流第一页；若帖子疑似多楼层正文或目录不完整，继续拉取楼主流后续页。
  2. 从首楼提取链接目录，得到 `pid -> title`。
  3. 从首楼提取纯文本目录，得到 `index -> title`。
  4. 扫描楼主楼层前若干短行，识别章节标题候选。
  5. 对每个明显有内容的楼主楼层生成章节候选，按 `pid` 去重并保持楼层顺序。
  6. 标题优先级：链接目录标题 > 楼层标题 > 纯文本目录顺序补名 > `第 N 段` / `未命名章节`。

- **保全式降级**：
  - 多楼层帖即使没有任何可识别标题，也要把楼主每个正文楼层作为章节或段落进入阅读流。
  - 单楼超长帖保持整楼正文完整，再做可选内部小节导航；内部拆分失败时仍显示整楼。
  - 低置信度筛除只用于降低入口权重和提示用户，不用于静默丢弃内容。
  - 对疑似公告、目录维护、作者说明的楼层，可标记为"说明"或弱化显示；只有非常明确的空楼、纯外链占位、重复目录才可排除。

- **完整性自检**：
  - `ReadingBook` 构建时记录 diagnostics：`opPostCount`、`chapterCount`、`linkedTocCount`、`plainTocCount`、`titleDetectedCount`、`fallbackChapterCount`、`droppedPostCount`、`declaredLatestNo`、`confidence`。
  - 当章节数明显少于楼主正文楼层数、标题声明进度或目录数量时，阅读器提示"已尽量保留楼主内容，章节标题可能不完整"。
  - `droppedPostCount` 应默认接近 0；只有命中强排除规则时才增加，并保留调试日志。

### 3.2 翻页技术：**方案 A — WebView + CSS 多栏**
- 每章 HTML 注入一个 CSS 多栏（`column-width:100vw` + 横向 `scroll-snap`）页面，**左右平移翻页**。
- 真分页重排、翻页动画、字号缩放、护眼主题、图片/引用渲染由 CSS 处理；正文本就是 HTML，几乎零转换。
- **不做 3D 卷页动画**（性价比低；微信读书默认也是平移）。
- 顶/底工具栏、目录抽屉用 **App 原生组件浮在 WebView 之上**，保持全 App 视觉一致；
  仅正文区是 WebView，靠 `postMessage` 桥接主题色、翻页进度、唤出工具栏的点击、图片点击。
- 否决方案 B（原生 `onTextLayout` 分页）：需自处理图片高度、改字号重排、长篇性能，坑多工作量大。

### 3.3 交互
- **入口**：帖子详情页提供「阅读模式」入口；判定为小说类时更突出提示。低置信度但仍可能有正文时，不隐藏入口，改用「尝试阅读模式」文案。
- **沉浸阅读**：默认隐藏工具栏；点击屏幕 左 1/3 上一页、右 1/3 下一页、中间 1/3 唤出/收起工具栏；左右滑动翻页；章末自然进入下一章。
- **顶部栏**：返回、书名/章节名、目录入口。
- **底部栏**：进度（第 N 话 / 百分比，可拖动跳转）、字号、主题。
- **识别提示**：当 diagnostics 显示章节标题不完整或存在兜底章节时，在首次进入或目录顶部显示温和提示，不阻断阅读。

### 3.4 v1 设置项（四项全做）
- 字号调节（实时重排）
- 护眼 / 主题底色（light/dark 之外加米黄等阅读主题）
- 阅读进度记忆（本地，复用 `history.ts` 范式，支持续读）
- 章节目录抽屉（列全部章节、高亮当前、点击跳转）

### 3.5 章末评论
- 每章正文结束后，附该章对应的读者评论（折叠/独立区块）。
- 实现：按 `pid → page` 取**未过滤**的那一页，楼主该楼当正文、其后到下一个楼主楼之间的读者楼当作"本章评论"，
  按需懒加载，避免预拉全部页。

### 3.6 正文内元素
- 图片：随正文排版；点击桥接回 App 现有原生 `ImageViewer`（放大/相册滑动）。
- 链接：站内 `tid` 跳转走现有逻辑；站外用系统浏览器打开。引用框正常渲染。

### 已知风险
1. WebView 手势与 RN 手势边界：翻页都在 WebView 内处理，仅"唤工具栏"的点击 postMessage 给 RN。
2. 正文走系统 web 字体，与 App 原生字体可能细微差异（阅读区独立沉浸界面，可接受）。
3. 章末评论"哪条评论属于哪章"按楼层位置近似归类，连载中夹楼的边界需调。
4. 571902 类无标记单楼帖的章节细分策略需进一步定；无论拆分是否成功，都必须保留"整楼一章"兜底。
5. 保全优先会让少量作者说明进入阅读流，需要用 UI 标记和后续规则优化，而不是用高风险过滤提前删除。

## 内容形态（决定阅读模式要适配的三种情况）
论坛帖子的小说正文有三种排布，阅读模式要统一呈现为"按章节阅读"：
1. 长篇连载（带目录链接）：楼主首楼有自动生成的目录，列出几百话，每话是楼主后面的
   一个独立楼层。
2. 中篇连载（目录无链接）：首楼有纯文本目录（第1话、第2话…），章节同样是楼主逐楼发布。
3. 短篇（无目录）：整篇正文都在首楼一个楼层里。
阅读模式统一抽象为：一本书 = 若干章节；正文只取楼主本人的内容，读者回复不进入正文流。
核心原则是内容保全：宁可多带一点楼主说明，也不能漏掉可能的正文。目录和标题用于整理内容，
不应成为丢弃楼主后续楼层的理由。

## 要设计的界面与交互
### 1. 入口
在现有「帖子详情页」上提供进入「阅读模式」的入口（按钮/动作）。当帖子被判定为小说类
时更突出地提示用户"用阅读模式打开"。请设计这个入口的位置与呈现。

### 2. 阅读主界面（沉浸式）
- 全屏沉浸：默认隐藏所有工具栏，屏幕主体只有正文。
- 正文按"一页填满一屏"的方式呈现，左右平移翻页（非上下滚动、非 3D 卷页）。
- 章节之间有清晰的章节标题分隔。

### 3. 翻页与点击区
- 左右滑动翻页（上一页/下一页）。
- 点击屏幕：左侧区域=上一页，右侧区域=下一页，中间区域=唤出/收起工具栏。
- 翻到一章最后，能自然进入下一章。

### 4. 工具栏（唤出时浮现，收起时消失）
- 顶部栏：返回、书名/章节名、目录入口。
- 底部栏：阅读进度（当前第几话 / 百分比，可拖动跳转）、字号调节入口、主题切换入口。

### 5. 目录抽屉
- 从工具栏打开章节目录（侧拉/弹出），列出全部章节标题，高亮当前章，点击跳转到任意章。
- 长篇有几百话，目录要能流畅浏览。
- 当章节由无标题楼层兜底生成时，目录中使用「第 N 段」等中性标题；顶部可提示"章节名可能不完整，正文已按楼主楼层保留"。

### 6. 阅读设置
- 字号：多档切换，正文实时重排。
- 主题/护眼：在浅色、深色之外，提供护眼底色（如米黄）等阅读主题。
- 阅读进度记忆：自动记住读到的位置，下次进入续读（请体现"上次读到…续读"的呈现）。

### 7. 章末评论
- 每章正文结束后，附该章对应的读者评论（折叠/独立区块），不打断正文阅读；用户可展开查看。

### 8. 正文内元素
- 图片：随正文排版显示，点击可放大查看。
- 链接/引用：正常呈现，可点击。

### 9. 各种状态
- 加载中、续读提示、读到最后一章的"完结/待更新"提示、网络错误重试。
