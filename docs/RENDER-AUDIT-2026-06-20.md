# 帖子富文本渲染对照审计（2026-06-20）

## 范围与方法

- 当前应用：`http://localhost:8081/tabs/forum`（复用已打开页面，未新开服务）。
- 原始网页：`https://bbs.yamibo.com/forum.php?showmobile=yes`（复用已打开页面）。
- 数据源：通过已登录本地代理读取 Discuz mobile API 与原站 mobile HTML，对照当前 `src/util.ts` 的 `parseMessage()` 输出。
- 登录态：代理返回 `cola43`，可读取文学区、资源区等登录可见内容。
- 抽样规则：论坛首页展示的 9 个顶层板块，每个板块取第 1 页 5 个主题；优先普通主题，不足时用页面可见主题补足。
- 对照粒度：每个主题检查第 1 页可见楼层（最多 20 楼）的原站 HTML 与 app 解析块，重点看链接、图片、投票、引用、表格、附件、隐藏/折叠、行内样式。

限制：本轮没有修改代码。Chrome 扩展在最终读取 console 时不稳定；已确认两个既有标签仍存在，但最终 console error 读取未完成。Chrome 环境检查显示 Chrome 正在运行、Codex Chrome Extension 在 Default profile 已安装且启用。

## 总结

本轮覆盖 9 个板块、45 个主题。普通正文整体没有发现系统性大段丢字；主要问题集中在 Discuz 特殊结构的保真度：

- **投票组件缺失**：多个板块的投票主题在原站显示选项/结果，app 只显示正文说明。
- **部分图片附件被降级为附件块**：贴图区图楼里，原站直接显示图片，app 有时显示为 `[附件] 文件名`，图片未进入图片浏览体验。
- **表格结构缺失**：使用指南/资源帖中的表格被压平成普通文本，可读性下降。
- **引用结构经常降级**：原站引用框在 app 中常变成普通文本或普通链接，没有 quote 块样式。
- **行内样式全部降级**：粗体、字体颜色、字号等原站强调样式在 app 中均为普通文本。
- **链接总体已有改善，但仍有边界**：正文中的普通 URL / `<a>` 可解析；图片包裹链接、站内引用链接、部分复杂 HTML 中的链接数量仍少于原站。

## 主要发现

### P1：投票主题在 app 中缺少投票选项/结果

影响：用户无法看到原站投票结构，只能看到楼主正文，容易误解主题内容。

代表样本：

- 動漫區 `565439`：原站显示 2 个投票选项，app 只显示“在看漫画时突然想到这两种剧情，大家更喜欢哪种？”。
- 動漫區 `541450`：原站显示投票比例与票数，app 只显示问题文本。
- 海域區 `572624`、`572638`、`572051`、`566834`：均为投票帖，app 未显示选项。
- 遊戲區 `519597`：flowers 投票缺失选项/结果。

建议：将 `viewthread` 中 poll 相关字段映射为只读投票块；v1 可以先显示选项、票数/比例和“暂不支持投票”提示。

### P1：贴图区部分图片被当作附件，而不是正文图片

影响：图楼和漫画楼的核心内容在 app 中从图片降级为附件文件名，用户无法直接预览或进入图片浏览器。

代表样本：

- 貼圖區 `556755`：
  - `#4` 原站显示 2 张图片，app 显示 `[附件] 1.jpg`、`[附件] 2.jpg`。
  - `#5` 原站显示 4 张图片，app 显示 4 个附件文件名。
  - `#8`、`#11` 同类问题。

推测：部分楼层附件字段未带 `isimage: "1"` 或字段形态与预期不同，当前 `parseMessage()` 将其按普通附件处理。

建议：附件缺少 `isimage` 时，按扩展名和 mime/路径后缀补判图片；`.jpg/.jpeg/.png/.gif/.webp` 应进入图片块。

### P1：表格被压平成普通文本

影响：教程和资源说明里的对照表可读性明显下降。

代表样本：

- 使用指南 `520079`：黑名单脚本教程含表格，app 只输出连续文本。
- 使用指南 `535335`：用户组、阅读权限、积分对应表被压平。
- 資源交流區 `548330`：资源信息表被压平。

建议：先支持简单表格块，至少按行/列渲染为紧凑文本表；复杂表格可降级为“表格内容”卡片。

### P2：引用框结构经常降级

影响：回复上下文不清晰，引用与正文混在一起。

代表样本：

- 管理版 `572407 #13`：原站引用 `ckyy14 发表于 ...`，app 文本里能看到内容，但没有 quote 块样式。
- 動漫區 `561618 #15`、`541450` 多个回复：引用内容没有保持引用框。
- 遊戲區 `572363`、影視區 `570821`、资源区多个回复都有同类情况。

建议：扩展 `parseMessage()` 对 `<div class="quote"><blockquote>...`、移动端引用链接和 API 中的引用片段做统一识别。

### P2：行内样式全部降级为纯文本

影响：不一定丢字，但原站强调信息丢失；教程、公告、版规尤其明显。

代表样本：

- 管理版 `535406`、`498791`
- 使用指南 `506902`、`535335`
- 文學區 `572256`
- 影視區 `549089`

建议：不必完整复刻 Discuz 样式，但可以支持 `bold`、`color`、`small/large` 三类最常见 span，或者在 block 内保留 inline runs。

### P2：链接边界仍需补强

影响：普通 URL 已能识别，但复杂 HTML 中的站内跳转、图片包裹链接、教程大量链接仍有少量丢失或语义降级。

代表样本：

- 管理版 `572320 #1`：GitHub / 站内参考链接大体保留，但原站还含图片附件链接。
- 使用指南 `506902 #1`：BOOTH 教程大量图片包裹链接，app 保留正文链接和图片，但链接数量少于原站。
- 影視區 `549902 #1`：原站整合帖有多条站内/外站链接，app 数量接近但仍少。

建议：把“图片打开链接”和“正文可点击链接”分开统计；图片链接不必重复显示为文字链接，但站内普通链接应完整转为端内跳转或外部打开。

## 正常或可接受项

- 大多数普通正文、中文/日文/韩文混排、emoji 能进入 app 文本。
- 直接裸 URL 和多数 `<a href>` 已能生成 link block。
- 文學區 `572665`、遊戲區 `571945` 这类普通文本主题未发现明显正文错漏。
- 新增的隐藏/折叠提示在本轮样本中没有遇到明确反例；本轮样本没有系统覆盖隐藏/折叠帖。

## 后续建议

1. 优先修复“图片附件误判为普通附件”，这会直接影响贴图区核心阅读。
2. 给投票帖加只读 poll block，至少展示选项/结果。
3. 补 table 和 quote 的结构解析，先做到不混成一段正文。
4. 建立一个小型富文本 fixture 集：从本次样本中固定 8-10 个楼层 HTML/API payload，后续每次改 `parseMessage()` 用脚本比对 block 输出。
5. Chrome 插件最终日志读取不稳定，后续人工视觉验收时建议先重连或重启 Codex Chrome Extension。
