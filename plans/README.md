## Dependency notes

- 001 should land first because the other four plans change runtime behavior that typecheck alone cannot protect.
- 002 depends on 001 so session migration and expiry behavior have characterization tests.
- 003 and 004 both introduce list pagination metadata; whichever lands first should define the shared `ListResult` shape for the other.
- 005 depends on 001 so proxy allowlist behavior is covered by a local verifier.
- 006 depends on 005 because it extends the already-landed proxy allowlist work from helper URLs to the normal forwarding path.
- 007 should land before behavior-heavy API work because it gives network calls a bounded failure mode.
- 008 is independent after 001; it only changes BoardScreen stale-response handling.
- 009 is independent after 001; it changes API favorite lookup behavior and should keep ThreadScreen optimistic UI intact.
- 010 is independent after 001; it adds parser regression coverage before hardening `parseForumLink`.
- 011 should land before 012 because 012's message stale-response guard assumes message errors are already scoped per tab.
- 012 extends the request-version pattern already used by 008 from board lists to collections and messages.
- 013 is independent after 001; it wires the existing Jest suite into the release workflow.
- 014 is docs-only and can land at any time, but should land before handing future write-operation work to agents.
